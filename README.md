## 概要

### 要件・機能

デザインカンプをもとに、以下のインタラクションおよび表示の切り替えを実装しています。

- **レスポンシブ・流体レイアウト:** スマートフォン、タブレット、PC（1440px〜）の各環境において、画面幅に応じて縮尺が可変（Fluid）するレイアウト。極端な画面幅の際にも可読性を損なわないよう文字サイズの上限・下限を設定。
- **JavaScriptを用いたコンポーネント実装（Vanilla JS）:**
  - **ハンバーガーメニュー:** モバイル環境におけるナビゲーションの開閉制御。状態に応じた `aria-expanded` の更新、画像アセット（`src` / `alt` 属性）の切り替え、および下層リンククリック時の自動クローズ処理。
  - **タブメニュー（Features）:** `role="tab"` / `role="tabpanel"` に基づき、`aria-selected` と `hidden` 属性を切り替えることでコンテンツを表示・非表示。セマンティックなHTML（button要素）の採用による、キーボードフォーカスが標準で当たるマークアップ構造。
  - **アコーディオン（FAQ）:** `scrollHeight` を用いた動的な高さ計算と、`requestAnimationFrame` を用いた開閉アニメーション。
  - **スクロールフェードイン:** `Intersection Observer API`（交差監視API）を利用した、スクロールに連動する要素のフェードイン演出。

## 開発プロセス
### ディレクトリ構造

Viteの静的アセット管理およびSassのコンパイル環境に準拠し、関心の分離（Separation of Concerns）を意識したディレクトリ設計を行っています。ローカル環境の依存ライブラリ（`node_modules/`）は `.gitignore` を用いてGit管理対象から除外しています。

```text
.
├── .gitignore               # Git管理除外設定（node_modules等の除外）
├── README.md                # 本ドキュメント
├── index.html               # メインHTML（エントリーポイントからJSを呼び出し）
├── package.json             # プロジェクト依存関係・各種スクリプト定義
├── package-lock.json        # 依存ライブラリのバージョンロック
├── preview.jpg              # README表示用のプレビュー画像
├── vite.config.js           # Vite設定ファイル
├── design/                  # 支給されたデザインカンプ・仕様イメージ（Git管理）
│   └── *.jpg                # 各画面・状態のレイアウト画像
├── public/                  # 成果物にそのままコピーされる静的アセット
│   └── images/              # ロゴ、アイコン、背景用SVG等
└── src/                     # ソースコードを集約する開発用ディレクトリ
    ├── main.js              # 全体統合用のメインJS
    ├── css/
    │   └── style.scss       # 各種状態定義・コンポーネントを包括するSCSS
    └── js/
        └── script.js        # DOM操作・アクセシリティ属性・UIアニメーションを制御するVanilla JS
```

### 使用技術・環境
- 言語・マークアップ: HTML5（WAI-ARIA準拠） / SCSS (Sass) / Vanilla JS (ES6+)
- リセットCSS: destyle.css (npm管理・SCSS側でのインポート)
- ビルドツール: Vite
- 環境自動化: GitHub ActionsによるGitHub Pagesへの自動CI/CDパイプライン構築

### こだわった技術的アプローチ
**1. vwとメディアクエリを組み合わせた流体レスポンシブ設計（SCSS）**

ブレイクポイントでの唐突なレイアウト崩れを防ぎ、画面幅のスムーズな縮尺変化（Fluid）を実現するため、vw 単位をベースにスタイルを設計しました。

ただし、vw 単体ではウルトラワイドモニターや極小画面において文字サイズや余白が極端な大きさになるリスクがあるため、主要なブレイクポイントごとに適切な縮尺になるよう数値を調整し、どのデバイスでも一定の視認性と可読性を保てるよう実装しています。

**2. WAI-ARIA属性を用いたステート管理（JS × SCSS）**

見た目（CSS）の切り替えだけでなく、支援技術やキーボード操作を行うユーザーへの配慮として、JavaScriptによる状態管理を行いました。

ハンバーガーメニュー開閉時の aria-expanded の更新や、タブ切り替えUIにおける aria-selected と hidden 属性の制御を行っています。スタイルをJS側で直接書き換えるのではなく、「JSは属性（状態）の切り替えのみを行い、スタイリングの変化はSCSS側がコントロールする」という設計を意識しました。

**3. アニメーション時の描画負荷低減とフォーカス制御（JS）**

ブラウザの描画負荷を抑え、実務で発生しやすいUI上の課題を回避するために以下の処理を行っています。

- **非表示コンテンツのTabキーフォーカス対策:** アコーディオンが閉じている際、目に見えない内部要素にTabキーのフォーカスが当たってしまうのを防ぐため、アニメーション完了（transitionend）時に hidden = true を付与し、フォーカス対象から除外されるよう制御しました。

- **描画タイミングの制御:** アコーディオンの開閉時には、scrollHeight を取得した直後に requestAnimationFrame を挟むことで、ブラウザの描画プロセスに合わせたスムーズなアニメーションとなるよう実装しています。また、処理を終えた transitionend のイベントリスナーは removeEventListener でその都度クリーンアップしています。

**4. 交差監視APIによるスクロール連動演出（Vanilla JS）**

スクロール連動のフェードイン演出（.fade-in）を実装するにあたり、window.addEventListener('scroll') によるイベントの過剰な発火を避けるため、IntersectionObserver を採用しました。

要素が画面内に入った瞬間（threshold: 0.1）に .is-active クラスを付与し、同時に observer.unobserve(entry.target) によって即座に監視を終了させることで、不要なイベント発火を抑える設計にしています。

### 今後の拡張計画
**JavaScriptのコンポーネント・モジュール化（ES Modules）**: 現状は script.js 内にすべてのUIロジックが集約されているため、今後のメンテナンス性向上を見据え、各UIロジックを独立したモジュールとして分割管理（「Tab.js」「Accordion.js」など）するリファクタリングを計画しています。その際、WAI-ARIA（Tabsパターン）で推奨されている「左右矢印キーによるタブのアクティブ制御」などのキーボード操作の拡張も合わせて行う予定です。
